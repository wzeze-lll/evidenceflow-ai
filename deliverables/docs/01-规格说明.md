# EvidenceFlow AI（证流 AI）产品规格说明

> 版本：1.0.0-alpha | 日期：2026-07-09 | 状态：Phase 1 开发阶段

---

## 一、项目背景与目标

### 1.1 项目背景

在当今信息爆炸的时代，学生、研究者和知识工作者每天需要处理大量文档资料。面对海量信息，如何快速提取关键内容、识别不同来源之间的异同、追踪每个结论的依据来源，是一个普遍存在却未被很好解决的问题。

现有工具存在以下局限：
- **通用 AI 聊天工具**（如 ChatGPT、DeepSeek）能回答问题，但无法自动引用原文出处，用户无法验证结论的真实性。
- **传统文档管理工具**（如 Notion、Evernote）擅长存储和检索，但缺乏 AI 分析能力。
- **学术文献管理工具**（如 Zotero）侧重于引用管理，但不具备多文档智能分析的能力。
- **企业级数据分析平台** 价格昂贵，需要部署服务器，不适合个人和小团队使用。

EvidenceFlow AI（证流 AI）正是为解决这一痛点而生。它是一款浏览器端的纯前端应用，专注于多文档智能分析，核心差异化优势在于：**每一个 AI 结论都附带可追溯的原文证据**。

### 1.2 项目目标

- **透明可信**：建立从文档到结论的完整证据链，让每个 AI 分析结果都有原文依据。
- **高效分析**：支持同时分析多份文档，自动发现冲突与共识，生成结构化决策简报。
- **隐私优先**：所有文档在浏览器本地解析处理，不上传至任何服务器。用户拥有完整的数据控制权。
- **轻量易用**：无需安装、无需注册、无需部署服务器。打开浏览器即可使用。

### 1.3 产品定位

EvidenceFlow AI 是一款面向学生、项目团队和研究人员的**浏览器端多文档智能分析平台**。它不是通用 AI 聊天工具，而是专注于"从材料中找答案"的垂直分析工具。

## 二、市场痛点与目标用户

### 2.1 三类目标用户及其痛点

#### 群体 1：学生（大学生、研究生）

| 场景 | 痛点 | EvidenceFlow 的解决方案 |
|------|------|------------------------|
| 写论文查资料 | 读了 10 篇文献，记不清哪个观点来自哪篇 | 证据链自动关联原文出处，点击即可查看 |
| 期末复习 | 多本教材和笔记中的知识点散乱 | AI 问答直接定位相关段落，支持跨文档搜索 |
| 小组项目决策 | 不同成员提供不同方案，不知道选哪个 | 冲突雷达对比各方案优劣，决策简报汇总结论 |

#### 群体 2：项目团队（产品经理、技术负责人）

| 场景 | 痛点 | EvidenceFlow 的解决方案 |
|------|------|------------------------|
| 技术方案评审 | 两份方案各有优劣，缺乏系统比较 | 共识地图展示双方一致的方面，冲突雷达聚焦分歧 |
| 需求文档分析 | 多个需求文档内容重叠、矛盾 | AI Reader 跨文档问答，自动标注矛盾点 |
| 项目复盘 | 总结报告需要引用多份过程文档 | 决策简报一键生成，附带原文引用 |

#### 群体 3：研究工作者（独立研究员、咨询师）

| 场景 | 痛点 | EvidenceFlow 的解决方案 |
|------|------|------------------------|
| 竞品调研 | 需要对比多份竞品分析报告 | 多文档对比分析，自动提取关键维度数据 |
| 行业研究 | 多份报告结论不一致，难判断可信度 | 冲突雷达标注高风险矛盾，建议核实步骤 |
| 快速出报告 | 老板要一份决策建议，没时间写 | 决策简报支持 5 种目标和 4 种受众级别 |

## 三、核心使用场景

### 场景 1：文献综述与论文写作

1. 上传 10 篇 PDF 论文到工作区
2. 在 AI Reader 中提问"各篇论文对深度学习模型优化有什么共同观点？"
3. 查看 Consensus Map，发现 3 篇论文一致认为数据增强是有效方法（强共识）
4. 查看 Conflict Radar，发现 2 篇论文对 batch size 的最佳值存在不同结论（中风险冲突）
5. 点击每个冲突的引用，跳转到原文查看具体数据和实验条件
6. 生成 Decision Brief，自动整理各论文的共识、分歧和建议

### 场景 2：技术方案选型评审

1. 上传需求文档、方案 A、方案 B、风险评估报告共 4 份文档
2. AI Reader 提问"方案 B 相比方案 A 有哪些优势？"
3. 系统返回带引用的回答，每个观点都标注来源文档和页码
4. Conflict Radar 识别出开发周期和架构选型的核心分歧
5. 生成决策简报，输出 9 个结构化章节：问题定义、关键事实、共识、争议、方案比较、风险评估、信息缺口、建议方案、行动计划

### 场景 3：快速知识查找

1. 上传课程 PPT（转 PDF）、教材电子版、个人笔记
2. 按 Cmd+K 打开命令面板，快速切换到 AI Reader
3. 输入"第三章的核心概念是什么？"，AI 从教材中找到相关段落
4. 点击引用编号，Evidence Drawer 弹出完整的原文出处

## 五、非功能需求

### 5.1 性能要求

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 首次加载时间 | < 3 秒（4G 网络） | Lighthouse Performance 评分 > 80 |
| 页面切换时间 | < 500ms | React Router 懒加载 + Suspense |
| AI 响应时间 | < 10 秒 | 从发送问题到收到完整回答的时间 |
| 文档解析速度 | 50MB 文件 < 30 秒 | 本地解析，不含上传时间 |
| 并发用户支持 | 100 用户 | 纯前端无后端，不受并发约束 |
| 索引数据库查询 | < 50ms | IndexedDB 本地查询 |

### 5.2 安全要求

| 要求 | 实现方式 |
|------|----------|
| 本地文档处理 | FileReader + 浏览器端解析，文件不离开设备 |
| API Key 保护 | 存储在浏览器 IndexedDB，不发送到任何第三方 |
| AI 请求脱敏 | 仅发送经过检索筛选的文本片段，不发送完整文档 |
| XSS 防护 | React 默认转义 + TypeScript 类型安全 |
| 无用户系统 | 不需要注册/登录，无密码泄露风险 |
| CSP（内容安全策略） | 无外部脚本动态加载 |

### 5.3 兼容性要求

| 浏览器 | 最低版本 | 备注 |
|--------|----------|------|
| Chrome | 90+ | 推荐使用 |
| Safari | 15+ | macOS/iOS |
| Firefox | 90+ | 桌面端 |
| Edge | 90+ | Chromium 内核 |

不支持：IE 11（缺少 IndexedDB 和 ES2023 特性支持）

### 5.4 隐私要求

- **Local-First 架构**：所有用户数据存储在浏览器 IndexedDB，不上传任何服务器。
- **无需注册**：不收集用户身份信息。
- **数据可清除**：设置页面提供一键清除所有本地数据功能。
- **无埋点追踪**：不集成任何分析 SDK 或追踪脚本。

### 5.5 可用性要求

- **响应式设计**：桌面端为主要目标平台，平板和手机端基本可用。
- **中文优先**：默认语言为中文，UI 文案和 Demo 数据均为中文。
- **无障碍访问**：键盘导航（Tab、Enter、Escape）、屏幕阅读器友好（ARIA 标签）、减少动画模式支持。
- **操作反馈**：所有操作有明确的加载状态、成功/失败提示。

## 六、范围边界

### 6.1 Phase 1 明确不包括的内容

| 类别 | 不包括的内容 | 原因 |
|------|-------------|------|
| 后端服务 | 无服务器、无数据库、无 API 服务 | 项目定位为纯前端应用 |
| 用户系统 | 无注册、登录、权限管理 | 数据本地存储，无需多用户 |
| 实时协作 | 无多人同时编辑 | Phase 1 聚焦单人使用 |
| 云存储 | 无文件云端同步 | 尊重隐私优先原则 |
| 移动端 App | 无 iOS/Android 原生 App | 通过移动浏览器访问 |
| 付费订阅 | 无付费墙、无功能限制 | 开源项目，免费使用 |
| 高级 PDF 渲染 | 无 PDF 原文渲染（复杂排版/图表） | 解析到文本层面，不做像素级渲染 |
| OCR | 无扫描件文字识别 | 需要服务端或大型模型支持 |
| 第三方集成 | 无 Slack/飞书/钉钉集成 | Phase 1 聚焦独立使用 |

### 6.2 已知限制

1. **PDF 解析精度**：对复杂排版（多栏、表格、公式）的 PDF 文件，文本提取效果有限。建议用户优先使用 Markdown 或 TXT 格式。
2. **大文件处理**：单文件上限 50MB。超大文件会导致浏览器内存压力。
3. **离线能力**：Demo 模式下可离线使用。使用远程 AI 时需联网。
4. **AI 回答质量**：依赖于所配置的 AI Provider 的能力。Mock Provider 仅提供演示品质的回答。
5. **存储限制**：受浏览器 IndexedDB 配额限制（通常几百 MB 到几 GB），超出后需清理数据。

## 七、术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| 工作区 | Workspace | 一个独立的分析项目环境，包含文档集合和分析结果 |
| 文档分块 | Document Chunk | 将长文档按章节或固定长度切分的文本片段，用于检索和引用 |
| 证据链 | Evidence Chain | 从 Claim（主张）到 Evidence（证据）再到原文的完整引用路径 |
| 引用 | Citation | AI 回答中的来源标注，包含文档名、页码、原文摘录和关系类型 |
| 冲突雷达 | Conflict Radar | 自动检测多文档之间观点矛盾的功能模块 |
| 共识地图 | Consensus Map | 展示多文档之间共识主题的可视化功能模块 |
| 决策简报 | Decision Brief | 结构化的分析报告，包含问题定义、关键事实、建议等章节 |
| AI Provider | AI Provider | 提供 AI 能力的服务接口抽象，支持多种后端（OpenAI、DeepSeek 等） |
| 知识卡片 | Knowledge Card | 从文档中提取的关键知识点，用于复习和记忆 |
