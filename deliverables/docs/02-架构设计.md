## 2. 技术选型理由

### 2.1 为什么选择 Vite 而不是 Next.js

**选择 Vite 的理由：**
- 本项目为纯客户端 SPA，无需 SSR/SSG
- Vite 开发服务器启动极快（<100ms HMR）
- 构建产物为纯静态文件，可部署到任何静态托管平台
- 不需要 Next.js 的服务端能力和额外的学习成本
- 项目规模适合 Vite 的简单配置

**为什么不选 Next.js：**
- Next.js 的 SSR 能力在本项目中无实际需求
- 增加不必要的复杂性和构建时间
- IndexedDB 为浏览器端存储，与 Next.js 服务端渲染存在天然冲突

### 2.2 为什么选择 IndexedDB/Dexie.js 而不是 PostgreSQL/MySQL

**选择 IndexedDB 的理由：**
- 核心数据流完全在浏览器端完成：文件解析 → 分块 → 本地检索 → AI 调用
- 天然支持离线使用，无需网络连接
- 零服务器成本，零运维负担
- 隐私优先：用户数据不离开浏览器
- Dexie.js 提供 Promise-based API，开发体验接近 MongoDB

**为什么不选传统数据库：**
- 引入 PostgreSQL/MySQL 需要后端服务，增加架构复杂度
- 用户文档数据敏感，本地存储更符合隐私要求
- 项目第一阶段无多设备同步需求
- 浏览器端存储已满足所有 Must Have 功能

### 2.3 为什么选择 Zustand 而不是 Redux

- Zustand API 极简，学习成本低
- 无 boilerplate，无 Provider 嵌套
- 与 TypeScript 配合良好
- 支持 middleware（persist、devtools）
- 包体积远小于 Redux + Redux Toolkit

### 2.4 为什么选择 Tailwind CSS v4

- v4 使用 CSS-first 配置，零 JS 配置文件
- 原子化 CSS 减少样式冲突
- 内置暗色模式支持
- 与 shadcn/ui 模式完美配合
- 构建时 Tree-shaking，只生成使用的样式

### 2.5 为什么选择 Mock Provider 作为默认

- 确保无 API Key 时应用仍可完整演示
- 降低课程答辩风险
- Mock 数据具有真实业务含义
- Provider 模式使真实 API 接入只需配置

### 3.3 设计原则

1. **关注点分离**：UI 组件只负责渲染，Service 层处理业务逻辑
2. **Provider 抽象**：AI Provider 通过统一接口接入，页面不直接调用 API
3. **数据流单向**：User Action → Store → Service → DB/API → Store → UI
4. **本地优先**：所有操作优先在本地完成，远程调用仅为可选增强

```
  │
  ▼

  │
  ▼

```

## 5. 数据模型

### 5.1 核心实体关系

```
Workspace 1 ──── * Document
Document 1 ──── * DocumentChunk
Document 1 ──── * AIConversation
AIConversation 1 ──── * AIMessage
AIMessage 1 ──── * Citation
Citation * ──── 1 DocumentChunk
Claim 1 ──── * Evidence
Evidence * ──── 1 Citation
AnalysisProject 1 ──── * ConflictItem
AnalysisProject 1 ──── * ConsensusTopic
AnalysisProject 1 ──── * DecisionBrief
DecisionBrief 1 ──── * BriefSection
```

### 5.2 存储方式

本项目使用 IndexedDB 作为唯一数据存储。Dexie.js 提供简洁的 Promise-based API。

**为什么选择 IndexedDB 作为唯一数据存储：**

1. **数据流闭环**：文档上传 → 本地解析 → 本地分块 → 本地检索 → AI 调用 → 本地缓存。整个流程在浏览器端形成闭环。
2. **隐私优先**：用户文档不离开设备，符合需求文档中"所有文件处理必须在用户设备本地完成"的要求。
3. **离线能力**：IndexedDB 使历史分析结果离线可查。
4. **零运维成本**：无需管理数据库服务器、备份、迁移。
5. **容量充足**：IndexedDB 通常有数百 MB 到数 GB 的存储空间，远超本项目需求。

**第一阶段不引入关系数据库的原因：**
- 无多用户协作需求（每个用户数据独立存储）
- 无跨设备同步需求
- 无服务端计算需求
- 引入数据库会增加不必要的架构复杂度

## 6. AI Provider 接口设计

```typescript
interface AIProvider {
  readonly name: string;
  readonly id: string;

  testConnection(): Promise<{ ok: boolean; message: string; latencyMs: number }>;

  chat(messages, context?): Promise<{ content: string; citations: Citation[] }>;

  streamChat(messages, context?, onChunk?): Promise<{ content: string; citations: Citation[] }>;

  summarize(document, chunks, mode): Promise<{ content: string; citations: Citation[] }>;

  compareDocuments(documents, relevantChunks): Promise<{ content: string; citations: Citation[] }>;

  detectConflicts(documentIds, documents, chunks): Promise<ConflictItem[]>;

  generateConsensusTopics(documentIds, documents, chunks): Promise<ConsensusTopic[]>;

  generateDecisionBrief(params, documents, chunks): Promise<DecisionBrief>;
}
```

Provider 通过工厂函数创建，根据用户设置选择具体实现。页面组件只依赖 `AIProvider` 接口，与具体实现解耦。

## 8. AI 建议与人工确认

本项目开发过程中，AI 辅助工具的建议均经过以下确认流程：
1. AI 提供技术方案 → 人工评估可行性和适用性 → 采纳/修改/拒绝
2. AI 生成代码 → 人工审查类型安全、边界条件、性能影响 → 合并
3. AI 建议架构 → 结合项目实际需求判断 → 选择最简可行方案
4. 所有安全相关代码（API Key 处理、数据存储）均经过人工审查

架构设计遵循"最简可行"原则，避免过度设计。每个技术选择都有明确的理由，而非堆砌技术名词。
