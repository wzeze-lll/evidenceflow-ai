## 三、AI 工具使用情况

### 3.1 使用的 AI 工具

| 工具 | 用途 | 覆盖阶段 |
|------|------|---------|
| Claude Code | 全流程 AI 辅助编程，包括代码生成、调试、审查、文档撰写 | Phase 1-6 |
| Claude Agent（子代理） | 多文件并行创建（UI 组件批量生成）、多模块并行修复（TypeScript 错误批量修复）、多文档并行撰写 | Phase 2-6 |
| Claude Code /code-review skill | 代码审查 -- 对变更进行正确性检查、简化建议 | Phase 5 |
| Claude Code /simplify skill | 代码简化 -- 检测可复用的模式、简化逻辑 | Phase 5 |

### 3.2 工具使用决策说明

- **为什么主要使用 Claude Code**：Claude Code 支持直接在项目环境中执行操作（读文件、写文件、执行命令、查看构建输出），无需在聊天界面和 IDE 之间复制粘贴，开发效率显著提升。
- **为什么使用 Agent 并行**：UI 组件创建（13 个 Radix 包装组件）、页面创建（9 个页面）、TypeScript 错误修复（20+ 个错误）都是"独立任务"，Agent 可以并行执行，大幅缩短开发周期。
- **为什么使用代码审查 Skill**：自动化审查可以发现人工容易遗漏的问题（如未使用的变量、类型不匹配、安全风险），作为人工审查的补充。

## 四、AI 协作工作流

### 4.1 需求分析阶段（Phase 1）

**AI 的角色：需求分析师 + 产品顾问**

工作流程：
1. 向 AI 提供完整的项目需求描述（包括用户群体、核心场景、功能清单）
2. AI 分析需求并输出：
   - 按 Must Have / Should Have / Nice to Have 分类的功能清单
   - 用户画像和核心场景描述
   - 非功能需求建议（性能、安全、兼容性）
   - 项目范围边界建议（明确不包括什么）
3. 人工审核 AI 的输出，进行以下调整：
   - 调整需求优先级（某些 AI 认为"Nice to Have"的可能实际是"Must Have"）
   - 修正 AI 对使用场景的理解偏差
   - 补充 AI 无法知道的项目特定约束（如答辩要求、时间限制）
4. 最终形成 01-规格说明.md

**典型 Prompt 示例：**

```
你现在是一名资深产品经理和全栈工程师。请帮我梳理一个
多文档智能分析 Web 应用的需求规格说明。核心创新点有四个：
1. 证据链（Evidence Chain）-- 每个 AI 结论必须关联原文出处
2. 冲突雷达（Conflict Radar）-- 自动检测多文档之间的观点矛盾
3. 共识地图（Consensus Map）-- 可视化展示多文档共识主题
4. 决策简报（Decision Brief）-- 生成结构化决策报告

目标用户：学生、项目团队、研究工作者。
技术约束：纯前端（无后端）、本地优先（隐私保护）、
浏览器 IndexedDB 存储。
```

**AI 建议被人工修改的案例：**
- AI 建议添加用户注册/登录系统 → 人工否决（纯前端本地应用不需要）
- AI 建议使用向量数据库 → 人工降级为关键词检索（Phase 1 足够）
- AI 建议集成多个 AI Provider（包括 Claude、Gemini 等）→ 人工改为 OpenAI 兼容接口（通用性更好，减少维护成本）

### 4.2 架构设计阶段（Phase 1）

**AI 的角色：架构师 + 技术顾问**

工作流程：
1. 向 AI 提供需求规格说明，要求输出架构设计方案
2. AI 输出：
   - 完整技术栈列表和版本号
   - 每个技术选型的理由（为什么选 A 不选 B）
   - 系统架构图和模块划分
   - 数据模型设计（实体关系图）
   - 关键接口定义（AIProvider interface）
3. 人工审核 AI 的架构方案：
   - 验证技术栈在当前开发环境中是否可成功构建（执行 `npm create vite@latest` 验证）
   - 评估方案是否过度设计（如 AI 建议的 BFF 层、微服务等）
   - 确认模块划分的粒度是否合理
4. 最终形成 02-架构设计.md

**AI 推荐的方案 vs 人工选择的方案：**

| 维度 | AI 初始推荐 | 人工选择 | 理由 |
|------|------------|---------|------|
| 框架 | Next.js | Vite + React | 纯客户端 SPA 不需要 SSR |
| 数据库 | PostgreSQL | IndexedDB (Dexie.js) | 无后端，本地存储更符合隐私要求 |
| 状态管理 | Redux Toolkit | Zustand | API 更简洁，项目规模不需要 Redux |
| 样式 | Tailwind CSS v3 | Tailwind CSS v4 | v4 使用 CSS-first 配置，更现代 |
| 后端 | BFF 层 (Express) | 无后端 | 纯前端应用，所有逻辑在浏览器完成 |
| 部署 | Docker + K8s | 静态文件托管 (Vercel) | 纯静态文件，不需要容器化 |
| 认证 | JWT + OAuth | 无认证 | 本地存储，不需要用户系统 |

**关键架构决策记录：**

1. **纯前端架构** -- 经过多轮讨论，确定 Phase 1 不引入任何后端服务。所有数据处理在浏览器完成。
2. **IndexedDB 作为唯一存储** -- 放弃 IndexedDB + 服务端数据库的混合方案，简化架构。
3. **AI Provider 接口抽象** -- 定义统一的 `AIProvider` 接口，支持 Mock/OpenAI/DeepSeek/Custom 四种实现。
4. **Mock Provider 默认** -- 确保无 API Key 也能完整演示，这是答辩和 Demo 的核心需求。

### 4.3 开发阶段（Phase 2-4）

**AI 的角色：全栈开发工程师**

这是 AI 辅助最密集的阶段，涉及大量代码生成。

**工作流程（以 UI 组件创建为例）：**

1. 向 AI 提供组件需求描述（如 "创建一个 Button 组件，支持 variant 和 size"）
2. AI 生成组件代码（使用 Radix UI + Tailwind CSS + cva）
3. 人工审查：
   - 类型安全（TypeScript 编译是否通过）
   - 样式一致性（Tailwind 类名是否正确）
   - 无障碍性（ARIA 属性、键盘导航）
   - 边界条件（disabled 状态、loading 状态）
4. 运行 `npm run build` 验证
5. 将组件纳入组件库

**开发过程中使用的 Prompt 分类：**

**类型 A：组件生成 Prompt**
```
Create a Button component with Radix UI Slot for asChild support.
Use cva (class-variance-authority) for variant management:
- variants: default, secondary, outline, ghost, destructive, link
- sizes: sm, md, lg, icon
Support all standard button HTML attributes via forwardRef.
Use Tailwind CSS classes. Add JSDoc comments.
```

**类型 B：服务层生成 Prompt**
```
Create a document parser service that handles file uploads:
- Accept File objects (PDF, DOCX, TXT, MD)
- Parse to extract text, page count, word count
- Split into chunks by section headings (##, #)
- Use FileReader API (browser-side only)
- Return Document and DocumentChunk[] objects
- Handle errors gracefully with typed error states
```

**类型 C：Demo 数据生成 Prompt**
```
Create a realistic demo dataset for a project evaluation use case.
The workspace compares 2 technical proposals for a campus AI project.
Include:
- 4 documents (requirements doc, 2 proposals, risk assessment) in Chinese
- Realistic AI conversation with citations
- 3 conflict items and 5 consensus topics
- A complete decision brief with 9 sections
All content should be in Chinese with realistic business context.
```

**AI 生成代码的质量分析：**

| 代码类型 | 首次可用率 | 需要修改的常见问题 |
|---------|-----------|-------------------|
| UI 组件（Radix 包装） | ~85% | 样式细节调整、ARIA 属性补充 |
| 页面组件 | ~70% | 状态管理逻辑、错误处理、空状态 |
| 服务层代码 | ~65% | 类型断言、边界条件、错误分支 |
| Demo 数据 | ~80% | 数据一致性、时间戳格式 |
| 配置文件 | ~90% | 少数路径和版本号调整 |

### 4.4 调试阶段（Phase 5）

**AI 的角色：调试助手 + 编译器错误修复器**

工作流程：
1. 运行 `npm run build`（`tsc -b && vite build`）
2. 收集 TypeScript 编译错误列表
3. 将错误列表提供给 AI，AI 并行修复多个文件的错误
4. AI 修复后再次运行 build 验证
5. 重复直到零编译错误

**实际修复的典型错误类型和数量：**

| 错误类型 | 数量 | 典型修复方法 |
|---------|------|-------------|
| 未使用的 import | 8 | 删除未使用的 import 语句 |
| 类型不匹配 | 5 | 修正类型注解或添加类型断言 |
| 未使用的变量 | 3 | 删除或使用 `_` 前缀 |
| 可能为 null/undefined | 4 | 添加非空断言或条件判断 |
| 隐式 any 类型 | 2 | 添加显式类型注解 |
| 中文引号与 TS 字符串冲突 | 1 | 改用模板字面量（backtick） |
| Dexie 表类型与业务类型不匹配 | 1 | 使用 `as any` 断言（评审 Accept） |

**调试阶段的 AI 协作经验：**
- AI 擅长批量修复同一类错误（如所有文件的未使用 import 可以一次清理）
- AI 在类型推导方面的判断需要人工验证 -- 有时 AI 建议的类型断言可能导致运行时错误
- 编译错误清零后，还需要进行运行时验证（在浏览器中实际操作一遍核心流程）

### 4.5 审查阶段（Phase 5）

**AI 的角色：代码审查员**

工作流程：
1. AI 对全部源代码执行静态审查
2. AI 按模块输出审查报告表格（#、严重度、文件、问题、影响、解决方案、状态）
3. AI 将发现的问题分类为：Fixed（已修复）、Pending（待修复）、Accepted Risk（接受风险）
4. 人工逐项验证：
   - 问题是否真实存在
   - 严重度评估是否准确
   - 解决方案是否可行
   - "Accepted Risk" 的项目是否确实可接受

**审查中发现的关键问题示例：**

- **真实 AI Provider 未实现**（高严重度）：AI 审查发现 provider.ts 中 OpenAI/DeepSeek 的 case 是注释占位符。这是一个真实且高影响的问题，但由于 Phase 1 以 Demo 演示为主，被标记为 Pending 而非立即修复。
- **PDF 解析回退逻辑简陋**（中严重度）：AI 审查指出 extractsTextFromPDFBuffer 使用正则从二进制提取文本，对复杂 PDF 几乎无效。对于 Demo 场景（使用 Markdown 和简单 PDF），影响可控，标记为 Accepted Risk。
- **错误处理不统一**（中严重度）：AI 发现 6+ 处静默 catch 块。部分场景（如 Dashboard 数据加载失败）确实可以静默处理，但关键操作（如文档上传失败）需要向用户反馈。

### 4.6 文档阶段（Phase 6）

**AI 的角色：技术写作助手**

工作流程：
1. AI 读取全部源代码文件，理解项目结构和实现细节
2. AI 生成各文档的初稿
3. AI 间并行处理多份文档（通过 Agent 机制）

   - 补充 AI 无法知道的实际协作细节
   - 调整文档结构和内容侧重点
   - 确保文档中描述的功能与实际代码一致
   - 添加真实的 Prompt 示例和协作经验

## 六、AI 输出评估与人工修正

### 6.1 修正案例详解

**案例 1：从 Next.js 到 Vite 的架构决策**

- **AI 初始建议**：使用 Next.js App Router，支持 SSR/SSG，利用 Server Components 优化性能。
- **AI 的理由**：Next.js 是 React 生态最流行的框架，内置路由、图片优化、SSR 等。
- **人工评估**：查阅了 Next.js 和 Vite 的文档对比，考虑到项目是纯客户端应用（所有数据在 IndexedDB），SSR 没有实际价值，反而增加复杂度和构建时间。
- **最终决策**：选择 Vite + React Router。更简单的配置，更快的 HMR，纯静态文件产出。
- **验证方法**：分别用两者创建了最小的原型项目，对比启动速度和构建大小。Vite 的 `create vite` 在 30 秒内即可运行，Next.js 需要额外配置 5 分钟以上。

**案例 2：从向量数据库到 TF-IDF 关键词检索**

- **AI 初始建议**：使用浏览器端 Transformers.js 进行语义向量检索，配合余弦相似度排序。
- **AI 的理由**：语义检索比关键词匹配更准确，Transformers.js 可以在浏览器运行小型模型。
- **人工评估**：Transformers.js 需要加载约 50MB 的模型文件，首屏加载时间会远超 3 秒的性能目标。Phase 1 的场景是几十个文档分块，TF-IDF 完全够用。
- **最终决策**：Phase 1 使用 TF-IDF 关键词检索。AI 回答质量通过 prompt engineering 和上下文窗口管理来保证。语义检索降级为 Nice to Have（后续版本）。
- **实际效果**：在 Demo 数据（15 个分块）的检索测试中，TF-IDF 的 Top-5 召回率达到预期。

**案例 3：状态管理从 Redux 到 Zustand**

- **AI 初始建议**：使用 Redux Toolkit + RTK Query 管理全局状态和服务端数据。
- **AI 的理由**：Redux 是最成熟的状态管理库，RTK Query 可以简化 API 调用。
- **人工评估**：本项目没有"服务端数据"，所有数据来自 IndexedDB。Redux 的 action/reducer/store 模式对于仅两个 Store（AppStore 和 SettingsStore）来说过于重量级。
- **最终决策**：使用 Zustand。一个 `create()` 函数即可定义一个完整的 Store，类型推导自然，不需要 Provider 包裹。
- **对比数据**：AppStore 代码约 55 行 Zusand vs 预计 120+ 行 Redux。SettingsStore 约 55 行 vs 预计 100+ 行。

### 6.2 AI 输出的常见需要修正的模式

| 模式 | AI 倾向 | 人工修正 | 修正原因 |
|------|--------|---------|---------|
| 技术选型 | 选择最流行/最新的技术 | 选择最适合项目规模的方案 | 流行不等于适用 |
| 架构设计 | 倾向于"完整架构"（有后端、有缓存、有队列） | 简化为最简可行架构 | 避免过度设计，降低维护成本 |
| 错误处理 | 生成 try-catch 但不填充 catch 块 | 补齐错误处理和用户反馈 | 提升生产环境可靠性 |
| 状态管理 | 使用 useReducer + Context | 使用 Zustand | 减少样板代码，提升可读性 |
| CSS 类名 | 混用 Tailwind v3 和 v4 的写法 | 统一为 v4 的 CSS 变量语法 | 保持代码一致性 |
| 注释 | 生成英文注释 | 关键注释使用中文 | 团队协作语言 |

## 七、AI 辅助开发的效率变化

### 7.1 效率数据

| 任务 | 传统开发预估 | AI 辅助实际 | 效率提升 | 备注 |
|------|------------|-----------|---------|------|
| UI 组件创建（13 个） | 3-4 小时 | ~5 分钟（Agent 并行） | ~40x | 主要是 Radix 的包装模板代码 |
| 页面组件创建（9 个） | 8-12 小时 | ~15 分钟（Agent 并行） | ~30x | 复杂页面需要多轮修改 |
| TypeScript 错误修复（20+ 个） | 1-2 小时 | ~2 分钟（Agent 批量） | ~40x | 主要是机械性清理工作 |
| Demo 数据编写 | 3-4 小时 | ~10 分钟 | ~20x | AI 生成中文业务内容质量高 |
| 配置文件和脚手架 | 0.5-1 小时 | ~2 分钟 | ~15x | Vite/TS/Vitest/Tailwind 配置 |
| 文档撰写（5 份文档） | 8-10 小时 | ~20 分钟（Agent 并行） | ~25x | AI 生成后需人工补充 |
| 代码审查 | 2-3 小时 | ~15 分钟 | ~10x | AI 快速扫描，人工逐项验证 |
| 类型定义编写 | 1-2 小时 | ~3 分钟 | ~20x | 200+ 行的类型定义 |

### 7.2 效率提升的归因

1. **Agent 并行**是最大的效率提升来源 -- 独立任务（如创建 13 个 UI 组件）可以同时执行，而非串行。
2. **机械性代码生成**效果最好 -- UI 组件包装、类型定义、配置文件等有明确模式的代码，AI 一次性生成质量很高。
3. **中文内容生成**超出预期 -- Demo 数据中的中文业务内容（需求文档、方案对比、风险评估）AI 生成质量很高。
4. **批量错误修复**非常高效 -- 同一类错误（如清理未使用的 import）AI 可以批量处理。

### 7.3 时间分布变化

**传统开发模式下的时间分布（估算）：**
- 需求分析：10%
- 架构设计：10%
- 编码实现：50%
- 调试修复：15%
- 文档编写：15%

**AI 辅助模式下的时间分布（实际）：**
- 需求分析：20%（需要更多时间来验证 AI 的理解是否正确）
- 架构设计：15%（需要更多时间来评估 AI 建议的合理性）
- 编码实现：25%（AI 生成代码很快，但审查和修改需要时间）
- 调试修复：10%（AI 批量修复编译错误）
- 文档编写：5%（AI 批量生成初稿）
- AI 输出验证与修改：25%（新增的时间投入，但非常有价值）

## 八、遇到的局限性

### 8.1 AI 工具层面的局限性

1. **一次性完美率有限**：AI 生成的代码约 65-85% 可以直接使用，其余需要人工修改。UI 组件的准确率最高（85%），复杂页面逻辑的准确率最低（65%）。
2. **上下文窗口限制**：每次与 AI 交互的上下文有限（约 200K tokens）。对于大型项目，需要分模块进行，然后人工整合。
3. **项目记忆问题**：AI 在连续的对话中可能会"忘记"之前确认过的设计决策，需要在关键节点重新提供上下文。
4. **中文支持偶有问题**：AI 在生成中文内容时偶尔会出现引号冲突、编码问题等。

### 8.2 架构决策层面的局限性

1. **倾向于"流行"而非"适合"**：AI 推荐的技术栈倾向于当前最流行的选择，但不一定是最适合该项目的选择。需要人工从"项目需求"出发重新评估。
2. **难以理解项目特有的约束**：AI 不知道项目的"答辩演示"需求、不知道开发者的实际技术水平、不知道可用的开发时间。这些约束需要人工明确告知。
3. **过度设计倾向**：AI 倾向于提供"完整"的方案（包含后端、数据库、缓存、消息队列等），而实际项目需要的是"最简可行"方案。

### 8.3 代码质量层面的局限性

1. **错误处理不完整**：AI 倾向于生成 `try { ... } catch { }` 的空 catch 块，不会主动考虑错误恢复和用户反馈。
2. **边界条件覆盖不足**：AI 生成的代码在主流程上正确，但边缘情况（空数组、网络断开、文件损坏）需要人工补充。
3. **类型安全漏洞**：AI 有时会使用 `as any` 绕过类型检查，而不是从根源上解决类型不匹配问题。
4. **测试代码缺失**：AI 不会主动为生成的代码编写测试，需要人工提示。

## 九、经验总结与反思

### 9.1 成功经验

1. **"先定范围再动手"是最重要的原则**：在开始任何编码前，先用 01-规格说明.md 明确定义 Must Have / Should Have / Nice to Have。这份文档成为后续所有 AI 交互的"锚点"，有效防止了需求蔓延。

2. **Agent 并行机制是效率倍增器**：将独立的子任务分配给不同的 Agent 同时执行，是对 AI 辅助开发效率提升最大的技术手段。UI 组件创建（13 个组件同时生成）是这一策略的最佳实践。

3. **"先 Mock 再真实"的开发策略**：Mock AI Provider 让整个系统在没有外部依赖的情况下可以完整运行和测试。UI、状态管理、数据库、引用映射等所有模块都可以独立验证。真实 Provider 接入时可以独立实现和测试。

4. **TypeScript Strict Mode 是第一道防线**：在 AI 生成大量代码的情况下，TypeScript 的类型系统在编译时捕获了大量错误。这些错误如果在运行时才发现，修复成本会高得多。

5. **Prompt 的质量决定产出的质量**：越是具体的需求描述，AI 生成的代码质量越高。模糊的 Prompt 产生模糊的代码。"创建一个看起来不错的仪表盘"远不如"创建仪表盘页面，包含 4 个统计卡片、最近文档列表、快速操作入口、活动日志"效果好。

### 9.2 需要改进的地方

1. **测试应该在开发过程中同步编写**：本项目是先完成所有功能开发，最后阶段才写测试。更好的做法是在 AI 生成每个服务模块时，同步要求 AI 生成对应的测试用例。

2. **响应式设计可以更早考虑**：部分页面最初是为桌面端设计的，后续才添加响应式适配。在设计阶段就应该明确每个页面在移动端的布局方案。

3. **错误处理应该在生成时明确要求**：在 AI Prompt 中增加错误处理的要求（"对每个异步操作添加错误处理，在 catch 中至少记录 console.error 并设置 UI 错误状态"），可以避免后续大量的错误处理补充工作。

4. **文档应该在开发过程中持续更新**：五份核心文档直到项目后期才完成。更好的做法是在每个阶段结束时更新对应文档，避免后期集中赶工。

### 9.3 核心反思

1. **AI 辅助开发的核心价值不在"替代人写代码"，而在"加速重复性工作"**。AI 在创建 UI 组件模板、类型定义、配置文件等有明确模式的代码方面效率极高。但在需要创造性和深度领域理解的方面（如架构决策、用户体验设计、业务逻辑），人的判断仍然是不可替代的。

2. **人机协作的最佳模式是"AI 生成 + 人工审查修改"**。AI 生成初稿的速度是人类手动编写的 10-40 倍，但初稿的质量需要人工把关。将 AI 定位为"高效的生产者"而非"独立的决策者"，是让 AI 成为有效协作工具的关键。

3. **好的 Prompt 是 AI 协作效率的核心杠杆**。投入时间写好 Prompt（明确需求、提供上下文、设定约束条件、给出输出格式），远比快速发出模糊的 Prompt 然后反复修改来得高效。花 5 分钟写好一个 Prompt，可能节省 30 分钟的代码修改时间。

4. **不要为了使用 AI 而使用 AI**。对于简单、明确的代码修改（如改一个变量名、调整一个样式），手动修改可能比描述 Prompt 更快。AI 工具的价值在于处理高复杂度和高重复性的任务，而不是替代所有人工操作。

5. **项目成功的核心因素仍是人的决策**。技术选型、架构设计、需求理解和创新方向都需要开发者基于项目实际的判断。AI 可以提供建议和加速执行，但不能替代这些核心决策。一个设计不良的系统，即使所有代码都由 AI 高效生成，仍然是一个失败的工程。
