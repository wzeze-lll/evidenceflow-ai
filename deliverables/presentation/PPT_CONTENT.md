# PPT Content Outline -- 毕业设计答辩 PPT 内容大纲

EvidenceFlow AI -- 基于大语言模型的多文档证据分析与决策辅助系统

---

## Slide 1: 封面

**标题**: EvidenceFlow AI
**副标题**: 证流 AI -- 可验证的多文档证据分析与决策辅助系统
**Slogan**: 让每一个结论都有证据可循
**信息**: 毕业设计答辩 | 姓名 / 学号 / 专业 / 指导老师 / 日期

**演讲备注**: 简短自我介绍。说明选题来源于对现有 AI 文档工具"只给结论、不給证据"这一痛点的观察——AI 说方案 B 更好，但"为什么"和"证据在哪"这两个问题始终没有被很好解决。

---

## Slide 2: 为什么要做这个项目

**关键要点**:
- ChatGPT/Claude 等工具在日常工作中广泛用于阅读和总结文档
- 核心问题: AI 给的结论是否可信？依据在原文的哪一页、哪一段？
- 在方案评审、文献综述、合同审查等严肃场景，结论的"可验证性"比"生成速度"更重要
- 现有工具的引用功能往往只是一个笼统的"来源: xxx.pdf"，用户无法快速定位原文

**演讲备注**: 用一个具体例子引入——假如 AI 告诉你"技术方案 B 优于方案 A"，你会信吗？你一定会追问"为什么？证据在哪？"。这就是本项目的出发点。我们不是在做"又一个文档聊天工具"，而是在做"帮你做出可信判断的证据分析工具"。

---

## Slide 3: 用户痛点分析

**关键要点**:
- **痛点 1: 信任危机** -- AI 生成的结论没有可追溯的证据链，"看着有道理但不敢用"
- **痛点 2: 信息冲突** -- 多份文档之间存在矛盾观点时，人工逐一比对效率极低且容易遗漏
- **痛点 3: 隐私焦虑** -- 将敏感工作文件上传到 ChatGPT/第三方服务存在数据泄露风险
- **痛点 4: 决策困难** -- 读完了多份资料仍然难以形成结构化的决策建议，"然后呢？"

**演讲备注**: 这 4 个痛点覆盖了学生研究者、项目评审者和企业决策者三类典型用户。后面会展示 EvidenceFlow 如何逐一解决这些问题。重点强调这不是凭空想象的需求——每个做过方案评审、论文综述的人都会遇到。

---

## Slide 4: 现有方案的局限性

**关键要点**:
- ChatPDF / PDF.ai: 仅支持单文档对话，缺乏跨文档分析能力。引用功能不精确
- NotebookLM (Google): 能读多文档但不主动发现冲突，无法生成结构化决策报告
- ChatGPT + 手动上传: 每次需重新上传，无持久化，引用不可靠（经常"编造"原文）
- 传统工具 (Zotero/EndNote): 擅长文献组织管理，但没有任何 AI 分析能力

**对比表格**:
| 能力 | ChatPDF | NotebookLM | ChatGPT | Zotero | EvidenceFlow |
|------|---------|------------|---------|--------|--------------|
| 多文档分析 | 不支持 | 支持 | 需手动 | 不支持 | 核心功能 |
| 证据溯源 | 基础 | 基础 | 不可靠 | 手动 | 可点击精确定位 |
| 冲突检测 | 无 | 无 | 无 | 无 | 核心创新 |
| 共识分析 | 无 | 部分 | 无 | 无 | 可视化展示 |
| 决策简报 | 无 | 无 | 部分 | 无 | 结构化输出 |
| 隐私保护 | 云存储 | 云存储 | 云端 | 本地 | 本地优先 |

**演讲备注**: 这个对比表格是答辩的核心论据之一。要强调 EvidenceFlow 填补的市场空白：市面上没有一个工具能同时做"多文档分析 + 证据溯源 + 冲突发现 + 决策辅助"。我们不是要替代 ChatGPT，而是要解决 ChatGPT 做不到的那些事。

---

## Slide 5: 产品整体方案

**关键要点**:
- 产品定位: "帮你在多份资料中找到可信答案的证据分析工具"，不是又一个 AI 聊天工具
- 核心流程: 上传文档 --> AI 分析理解 --> 发现共识与冲突 --> 追踪证据链 --> 生成决策简报
- Slogan: "让每一个结论都有证据可循"
- 使用方式: 纯浏览器运行，无需安装、无需注册、无需上传文件到服务器

**演讲备注**: 简洁有力地展示产品全貌。四个步骤形成从"原始文档"到"可执行决策"的完整闭环。重点强调差异化——市面上的 AI 文档工具都在做 Step 1-2（读 + 聊），我们在做 Step 3-4（分析 + 决策）。

---

## Slide 6: 创新点 1 -- 证据链 (Evidence Chain)

**关键要点**:
- **核心机制**: 每个 AI 结论 (Claim) 绑定多条证据 (Evidence)，每条证据精确指向原文位置（页数、段落、章节）
- **四种证据关系**: Support（支持）、Contradict（反对）、Complement（补充）、Uncertain（不确定）
- **置信度评估**: High / Medium / Low / Unverified 四档，帮助用户判断结论可靠性
- **交互方式**: 三栏布局 -- 左侧 Claim 列表，中间证据卡片，右侧 Overview 统计面板。支持按关系类型筛选。点击证据卡片可查看全文引用详情

**演讲备注**: Evidence Chain 是产品最基础也是最核心的功能。展示截图，重点指出"点击证据卡片右侧滑出原文引用"这个关键交互，这是竞品做不到的。

---

## Slide 7: 创新点 2 -- 冲突雷达 (Conflict Radar)

**关键要点**:
- **核心机制**: 选择 2-5 份文档，自动检测不同文档之间的观点冲突，按严重等级和类型分类
- **五种冲突类型**: Data（数据）、Definition（定义）、Opinion（观点）、Timeline（时间线）、Methodology（方法论）
- **对比展示**: 每个冲突主题下并列展示不同文档的具体观点和原文证据，配 AI 分析评述和验证建议
- **Comparison Matrix**: 以矩阵表格形式直观展示所有文档在所有冲突话题上的立场分布

**演讲备注**: 冲突检测是本产品最大的差异化创新功能。用 Demo 中的真实案例说明：两份技术方案对于"开发周期"存在 12 周 vs 14 周的冲突，AI 不仅指出了这个冲突，还分析了可能原因并给出了验证建议——"建议要求提供详细 WBS 工时估算表"。

---

## Slide 8: 创新点 3 -- 共识地图 (Consensus Map)

**关键要点**:
- **核心机制**: 识别多文档之间达成共识的主题，按共识强度分四档：强共识、中等共识、弱共识、存在争议
- **量化展示**: 每个共识主题标注文档覆盖率百分比；展示支持文档的原文摘录
- **可视化**: Recharts 柱状图展示各主题覆盖率对比；Document x Topic 矩阵表格展示每个文档对每个主题的立场
- **价值**: 帮助用户区分"已确认信息（有多方佐证）"和"待验证信息（仅来自单一来源）"

**演讲备注**: 共识分析的价值在于帮助用户识别"可靠结论"。例如 Demo 中"React 作为前端框架"获得两个技术方案的一致选择——这可以直接作为决策依据。而"移动端不是第一阶段重点"仅来自一份文档——需要谨慎对待。

---

## Slide 9: 创新点 4 -- 决策简报 (Decision Brief)

**关键要点**:
- **核心机制**: 自动将多文档分析结果转化为结构化的决策简报
- **标准结构（9 个 Section）**: 问题定义 --> 关键事实 --> 主要共识 --> 核心争议 --> 方案比较 --> 风险评估 --> 信息缺口 --> 建议方案 --> 下一步行动计划
- **引用完整性**: 每个 Section 的关键结论都关联原文引用，保持从结论到原文的全链路可追溯
- **输出能力**: 支持 Markdown / TXT 格式导出、浏览器直接打印、一键复制全文

**演讲备注**: 决策简报是产品的"最终交付物"——它不是 AI 随意生成的摘要，而是一份有引用、有分析、有对比、有建议的结构化决策文件。展示 Demo Brief 的完整 9 段结构及底部的引用来源编号。

## Slide 10: 技术架构

**关键要点**:
- **前端框架**: React 19 + TypeScript 6.0 (Strict Mode) + Vite 8
- **状态管理**: Zustand (轻量化、无 Provider 嵌套、TypeScript 友好)
- **本地数据库**: IndexedDB via Dexie.js (13 个表，覆盖所有业务实体)
- **样式系统**: Tailwind CSS v4 (CSS-first theme via `@theme` + CSS 变量暗色模式)
- **UI 原语**: Radix UI (Dialog/Select/Tabs/Tooltip/DropdownMenu 等无样式原语)
- **AI 抽象层**: AIProvider 接口 + Mock Provider (默认) + 可扩展适配器模式
- **检索**: TF-IDF 关键词检索（中英文分词 + 多因子加权 + 短语匹配加分）
- **图表**: Recharts (共识覆盖率柱状图)
- **测试**: Vitest + @testing-library (jsdom 环境)

**演讲备注**: 用架构图配合讲解。重点强调两个设计理念：1) Local-First -- 一切在浏览器端运行，零后端依赖；2) Provider Pattern -- AI 接口抽象使得系统不绑定任何特定 AI 服务，Mock/Demo 和真实 API 使用同一套代码。

## Slide 12: 典型迭代与关键决策

**关键要点**:
- **3 个典型迭代示例**:
  1. TypeScript strict mode 适配: 40+ 类型错误逐一修复，最终零错误通过
  2. Tailwind CSS v4 迁移: 从 JS 配置文件迁移到 CSS 变量方案，暗色模式无缝切换
  3. 数据流重构: 从组件内硬编码 Mock 数据统一为 IndexedDB 读取，确保数据流一致性
- **3 个关键架构决策**:
  1. Vite 而非 Next.js -- 纯前端 SPA 不需要 SSR，Vite 更轻更快
  2. IndexedDB 而非 PostgreSQL -- 本地优先架构的核心选择，零运维成本
  3. Mock Provider 作为默认 -- 降低演示门槛，确保答辩环境万无一失

**演讲备注**: 这一页展示的是工程能力——不仅会写代码，还会做架构决策和问题诊断。强调：AI 倾向于推荐"流行"方案（如 Next.js），但我们需要自己判断什么方案"适合"。

## Slide 13: 代码质量与测试

**关键要点**:
- **类型安全**: TypeScript strict mode 全量启用 (strict: true / noUnusedLocals / noUnusedParameters / verbatimModuleSyntax)，零编译错误
- **代码检查**: Oxlint (Rust-based，比 ESLint 快 50-100 倍)，0 error / 2 warning
- **测试体系**: Vitest (jsdom 环境)，19 个测试用例，100% 通过。覆盖工具函数和 TF-IDF 检索算法
- **构建流程**: `tsc -b && vite build` 双保险 -- 先类型检查再打包构建
- **代码分割**: React.lazy + Suspense 按路由分割，Dashboard 保持 eager import 避免首屏闪烁

**演讲备注**: 强调"零 TypeScript 编译错误"和"测试全部通过"是代码质量的底线证明。展示 `npm run build` 和 `npm test` 的终端输出截图。
